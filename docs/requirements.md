## 1. プロダクト概要

### 1-1. 概要

| 項目 | 内容 |
|------|------|
| プロダクト名 | LLM-IME（仮称） |
| 対象OS | macOS 13以降 / Windows 10・11 |
| 実装言語 | Rust（コア） |
| コア差別化 | LLMによる文脈依存変換 + 右側文脈トリガー再推論 + 学習型個人辞書 + プライバシー透明性 |

### 1-2. プロダクトビジョン

既存IME（Google日本語入力・MS IME・ATOK）は変換速度を最優先に設計されているが、本プロダクトは**変換精度・個人適応性・プライバシー透明性**を主な訴求点とする。

- 文脈を考慮した同音異義語の高精度変換
- 右側文脈が生まれた段階で自動的に過去の変換を補正する
- ユーザーの変換傾向を蓄積・反映する個人最適化
- ライブ変換UXによるLLMレイテンシの透過的なマスク
- クラウド推論・ローカルLLMをユーザーが制御できる透明な設計

### 1-3. ターゲットユーザー

| ユーザー層 | 想定する利用形態 |
|-----------|----------------|
| 一般ユーザー | ハイブリッドモードで精度の恩恵を受けつつ、センシティブ入力は自動保護 |
| プライバシー重視ユーザー | プライバシーモード固定でクラウド送信ゼロ |
| ローカルLLMプロフェッショナル | 自前モデルをIMEに統合して完全オフライン高精度変換・右側文脈再推論を活用 |

### 1-4. スコープ

| 対象 | v1 | v2以降 |
|------|----|--------|
| 言語 | 日本語 | 中国語・韓国語 |
| OS | macOS / Windows | Linux |
| LLM推論 | クラウド（Workers AI）+ ローカルLLM + ローカルN-gram | — |
| 再推論 | 右側文脈トリガー・選択再変換 | 文字数変動を伴う再推論 |
| 個人化 | 傾向DB + リランキング | LoRAアダプタ更新 |
| 辞書同期 | なし | iCloud / OneDrive連携 |

## 2. 用語定義

| 用語 | 定義 |
|------|------|
| **ローカルN-gram** | Vibrato + KenLMによる変換。LLMを使用しない高速ローカル処理 |
| **クラウドLLM** | Cloudflare Workers AIによるリランキング。インターネット接続が必要 |
| **ローカルLLM** | ユーザーが所有するGGUFモデルをllama-cpp-rsで直接実行する推論 |
| **アダプティブ変換** | 確定済みトークン数に応じて変換エンジンを自動切替する仕組み |
| **ライブ変換** | 打鍵と同時にリアルタイムで変換結果を表示するUX |
| **傾向DB** | ユーザーの変換選択履歴を蓄積・参照するSQLiteデータベース |
| **リランキング** | N-gramが生成した候補リストをLLMがスコアリングし直す処理 |
| **閾値T** | ローカルN-gramとLLMモードを切り替える確定済みトークン数の境界値 |
| **入力モード** | ユーザーが選択するプライバシー / パフォーマンス / ハイブリッド / ローカルLLMの動作方針 |
| **プライバシーモード** | 常にローカルN-gram処理のみ。クラウド送信を一切行わない |
| **パフォーマンスモード** | 長トークン時にクラウドLLMを使用する |
| **ハイブリッドモード** | フィールド種別を自動判定し、プライバシー / パフォーマンスを自動切替する |
| **ローカルLLMモード** | ユーザー所有のGGUFモデルを使用する上級者向けモード |
| **センシティブフィールド** | パスワード・クレジットカード番号等、クラウド送信を禁止すべき入力欄 |
| **ステータスインジケーター** | ハイブリッドモード時にアクティブなサブモードを常時表示するUI |
| **推論抽象レイヤー** | クラウド・ローカルLLM・N-gramを透過的に切り替えるRust trait |
| **選択再変換** | 確定済みテキストを選択した際に、左右の文脈を使ってLLMで再推論する機能 |
| **右側文脈トリガー再推論** | 打鍵継続により右側文脈が生まれた形態素境界で、低確信度トークンを自動再推論する機能 |
| **確信度** | LLMがある変換候補を選択した際のスコア（0〜1） |
| **LoRAアダプタ** | ベースモデルに追加する軽量な個人適応パラメータ（数十MB） |

### 3-2. 推論抽象レイヤー

```rust
trait LLMInferenceBackend {
    // 通常リランキング
    async fn rerank(
        &self,
        context: &str,
        candidates: &[String],
    ) -> Result<Vec<RankedCandidate>>;

    // 再推論（左右コンテキストあり）
    async fn rerank_with_right_context(
        &self,
        left_context:  &str,
        target:        &str,
        right_context: &str,
        candidates:    &[String],
    ) -> Result<Vec<RankedCandidate>>;
}

struct CloudBackend;       // Cloudflare Workers AI
struct LocalLLMBackend;    // llama-cpp-rs
struct OllamaBackend;      // Ollama API
```

### 3-3. 非同期パイプライン

```
打鍵イベント（メインスレッド）
      │
      ├─→ ローカルN-gram変換（即時 ~5ms）→ UIに仮候補を表示
      │
      ├─→ LLMキュー（長トークン時）
      │     デバウンス50ms → LLMリランキング → UI更新
      │
      └─→ 再推論キュー（形態素境界検出時）
            低確信度トークン特定
            → バックグラウンドで右側文脈付き再推論
            → 更新条件を満たしたらUI更新（アニメーション）
```

## 4. 変換モード設計

### 4-1. 全モード共通の基底ルール

> **確定済みトークン数が閾値T未満の場合、入力モードに関わらず常にローカルN-gramを使用する。**

```
確定済みトークン数 < 閾値T  →  全モード共通: ローカルN-gram + 傾向DB
確定済みトークン数 ≥ 閾値T  →  入力モードに応じてエンジンを選択
```

### 4-2. 各入力モードとエンジンの対応

```
                        短トークン（< 閾値T）    長トークン（≥ 閾値T）
                   ┌────────────────────────┬──────────────────────────────────┐
プライバシーモード  │  ローカルN-gram          │  ローカルN-gram + 傾向DB          │
パフォーマンスモード│  ローカルN-gram          │  クラウドLLM                      │
ハイブリッドモード  │  ローカルN-gram          │  センシティブ → ローカルN-gram     │
                   │                        │  非センシティブ → クラウドLLM      │
ローカルLLMモード  │  ローカルN-gram          │  ローカルLLM（GGUF / Ollama）     │
                   └────────────────────────┴──────────────────────────────────┘
```

### 4-3. アダプティブ変換の閾値

- **閾値T デフォルト値**: 15トークン（設定で変更可）
- **ウォームアップ**: 閾値T-5トークンの時点でLLMクライアントを初期化
- **モード遷移時の整合性**: 遡り修正は行わず、新規変換からLLMモードを適用

### 4-4. スコアリング

#### 短トークン時（全モード共通）

```
変換候補スコア =
  N-gram確率           × 0.6
  + ユーザー傾向スコア  × 0.3
  + ユーザー辞書ボーナス × 0.1
```

#### 長トークン・LLMモード時（クラウド / ローカルLLM共通）

```
変換候補スコア =
  LLMリランキングスコア × 0.6
  + N-gram確率          × 0.2
  + ユーザー傾向スコア  × 0.15
  + ユーザー辞書ボーナス × 0.05
```

> **備考**: 上記重みは仮値。スパイクタスク S-001 にて実験的に調整する。

### 4-5. ライブ変換の挙動

- 打鍵のたびにN-gram候補をUIに即時反映（仮表示）
- 打鍵停止後50msでLLMリランキングを実行（長トークン時のみ）
- LLM結果が返り次第、候補リストをアニメーション付きで更新
- スペースキーによる明示変換も引き続きサポート（後方互換）

## 5. 入力モード設計

### 5-1. 4モードの定義

| モード | 短トークン | 長トークン | クラウド送信 | 推奨ユースケース |
|--------|-----------|-----------|-------------|----------------|
| **プライバシー** | ローカルN-gram | ローカルN-gram + 傾向DB | なし | 機密業務・オフライン環境 |
| **パフォーマンス** | ローカルN-gram | クラウドLLM | あり（長トークン時） | 精度優先・通常利用 |
| **ハイブリッド** | ローカルN-gram | フィールド判定で自動切替 | 非センシティブ時のみ | **デフォルト推奨** |
| **ローカルLLM** | ローカルN-gram | ローカルLLM（GGUF） | なし | ローカルLLMプロフェッショナル |

### 5-2. ハイブリッドモードの自動切替ロジック

```
テキストフィールドにフォーカス → センシティブフィールド判定
  ├── センシティブ検出  → プライバシーがアクティブ
  ├── 非センシティブ検出 → パフォーマンスがアクティブ
  └── 判定不能          → プライバシーがアクティブ（安全側）
```

### 5-3. センシティブフィールドの検出

#### macOS（Accessibility API）

| 判定対象 | 取得方法 | 検出可否 |
|----------|---------|--------|
| パスワードフィールド | `AXRole = AXSecureTextField` | ✅ |
| ブラウザの `input type="password"` | ブラウザがAXに公開 | ✅ |
| Electronアプリ独自実装 | — | ❌（判定不能扱い） |

#### Windows（TSF InputScope）

| 判定対象 | InputScope値 | 検出可否 |
|----------|-------------|--------|
| パスワードフィールド | `IS_PASSWORD` | ✅ |
| クレジットカード番号 | `IS_CREDITCARDNUMBER` | ✅ |
| ゲームエンジンUI | — | ❌（判定不能扱い） |

> 判定不能時は**必ずプライバシー側にフォールバック**する。

### 5-4. ステータスインジケーター

ハイブリッドモード選択時のみ表示する。

| 状態 | 表示 | 色 |
|------|------|----|
| フォーカスなし | アイコンのみ | グレーアウト |
| 非センシティブフィールドにフォーカス | ⚡ パフォーマンス | ブルー |
| センシティブフィールドにフォーカス | 🔒 プライバシー | アンバー |
| 判定不能フィールドにフォーカス | 🔒 プライバシー | グレー |

```

⚡ LM Studio / Jan のモデルを自動検出  [スキャン]
```

## 6. 再推論設計

### 6-1. 再推論の2種類

本プロダクトでは2種類の再推論機能を提供する。

| 機能 | トリガー | 対象ユーザー | v1対象 |
|------|---------|------------|--------|
| **右側文脈トリガー再推論** | 打鍵継続による形態素境界の発生 | 全ユーザー | ✅ |
| **選択再変換** | 確定済みテキストの選択操作 | 全ユーザー | ✅ |

---

### 6-2. 右側文脈トリガー再推論

#### 概念

```
【通常のライブ変換】右側文脈なしで変換
  「プロジェクトの」+ 「きかん」→ 「機関」（確信度: 0.55）

【右側文脈トリガー再推論】打鍵継続で精度が向上
  ユーザーが「を確認した」と打ち続ける
       ↓ 「を」の入力で形態素境界を検出
  左: 「プロジェクトの」  右: 「を確認した」
  → LLMが再推論 → 「期間」（確信度: 0.91）→ 自動更新 ✅
```

#### トリガー条件

```
以下をすべて満たす場合に再推論キューに投入する:

1. 新しい形態素境界が検出された
2. 直近Nトークン以内に確信度 < 閾値P のトークンが存在する
3. 対象トークンの右側に新たな形態素が1個以上追加された
```

#### 更新条件

```
再推論後のスコアが以下をすべて満たす場合のみテキストを更新する:

1. 再推論後の確信度が元の確信度を 閾値D 以上上回った
2. 変換後の文字数が変わらない（カーソル位置ズレを防止）
3. 現在ユーザーが入力中の文字に影響しない位置のトークンである
```

#### 再推論パラメータ

| パラメータ | 意味 | 暫定デフォルト値 |
|-----------|------|----------------|
| N | 再推論対象とする過去トークン数 | 5 |
| P | 再推論をトリガーする確信度の上限 | 0.7 |
| D | 更新をかける確信度の改善幅 | 0.2 |

> 上記はすべてユーザー設定で変更可能とする。

#### コスト制御

```
クラウドLLMモード時:
  形態素ごとに毎回APIリクエストを送ると多発するため
  複数形態素（3〜5個）をまとめて1リクエストで再推論する

ローカルLLMモード時:
  APIコストがないため形態素境界ごとに即時実行できる
  → この機能の恩恵を最も受けるのはローカルLLMモードユーザー
```

#### 更新時のUX

```
確信度の改善幅に応じてUXを切り替える:

改善幅が大きい（D ≥ 0.3）:
  → テキストをハイライトアニメーションで置換（ユーザーに通知）

改善幅が小さい（0.2 ≤ D < 0.3）:
  → サイレント更新（アニメーションなし）

改善幅が閾値D未満:
  → 更新しない
```

### 6-3. 選択再変換

#### 概念

```
確定済み: 「今日の会議でプロジェクトの[機関]を確認した」
                                        ↑ 選択

LLMへのクエリ:
  左コンテキスト: 「今日の会議でプロジェクトの」
  選択テキスト:   「機関」
  右コンテキスト: 「を確認した」  ← 通常ライブ変換より精度が高い

→ 候補ポップアップを表示してユーザーが選択
```

通常のライブ変換は右側文脈が存在しないが、選択再変換では左右両方の文脈を使えるため精度が向上する。

#### トリガー方式

| 方式 | 概要 | 採用 |
|------|------|------|
| インラインポップアップ | 選択後200msでポップアップを自動表示 | ✅ デフォルト |
| ショートカットキー | Cmd+Shift+R（再変換）でトリガー | ✅ 併用 |
| ダブルクリック | 単語単位選択と同時に再変換候補を表示 | 🟡 v2検討 |

#### 選択再変換のフロー

```
確定済みテキストを選択
  ↓ 200msデバウンス（ドラッグ選択完了を待つ）
ポップアップ表示 + バックグラウンドでLLM再推論開始
  ↓ ~300ms
候補リストが揃う → ユーザーが候補を選択 or Escで閉じる
  ↓
OS APIで選択範囲を置換（macOS: insertText / Windows: ITfRange::SetText）
  ↓
傾向DBに再変換結果を記録 → 次回以降の変換精度に反映
```

#### ポップアップ抑制条件

```
以下の場合はポップアップを表示しない:
  ・選択範囲が20文字を超える（コピー等の可能性が高い）
  ・選択範囲が英数字のみ
  ・選択直後にCmd / Ctrlが押された（コピー・カット操作の可能性）
```

### 6-4. 再推論と傾向DBの連携

```
再推論結果はすべて傾向DBに記録する

記録内容:
  ・読み
  ・元の変換（初回LLM結果）
  ・再推論後の変換（確定値）
  ・左右コンテキスト
  ・再推論トリガー種別（右側文脈 / 選択）

効果:
  再推論データが蓄積されるほど
  同じ文脈での初回変換精度が向上していく
```

## 10. データ設計

### 10-1. 傾向DB スキーマ

```sql
CREATE TABLE conversion_preference (
  id              INTEGER PRIMARY KEY AUTOINCREMENT,
  reading         TEXT NOT NULL,    -- 読み（例: きかん）
  surface         TEXT NOT NULL,    -- 表記（例: 機関）
  prev_context    TEXT DEFAULT '',  -- 直前N-gram
  next_context    TEXT DEFAULT '',  -- 直後N-gram（再推論で追加）
  confidence      REAL DEFAULT 0.0, -- 確信度
  reinfer_trigger TEXT DEFAULT '',  -- 再推論トリガー種別（right_context / selection / ''）
  count           INTEGER DEFAULT 1,
  last_used_at    DATETIME DEFAULT CURRENT_TIMESTAMP
);

CREATE INDEX idx_reading ON conversion_preference(reading);
CREATE INDEX idx_reading_context ON conversion_preference(reading, prev_context);
CREATE INDEX idx_reading_full_context ON conversion_preference(reading, prev_context, next_context);
```

- `next_context` が空 = 右側文脈なし（短トークン時の記録）
- `next_context` あり = 再推論データ → 次回以降の初回変換精度向上に活用

### 10-2. ユーザー辞書 スキーマ

```sql
CREATE TABLE user_dictionary (
  id         INTEGER PRIMARY KEY AUTOINCREMENT,
  reading    TEXT NOT NULL,
  surface    TEXT NOT NULL,
  pos        TEXT,
  priority   INTEGER DEFAULT 5,
  created_at DATETIME DEFAULT CURRENT_TIMESTAMP
);
```

### 10-3. データ保存場所

| OS | パス |
|----|------|
| macOS | `~/Library/Application Support/LLM-IME/` |
| Windows | `%APPDATA%\LLM-IME\` |

## 11. 実装ロードマップ

### ロードマップ策定の原則

| 優先基準 | 内容 |
|----------|------|
| 依存関係 | 土台なしに上が建てられないものを先行 |
| リスク早期解消 | スパイクタスクを前倒し |
| 動くものを早く | 各フェーズ末に実際に使える状態を維持 |
| 外部依存の少ない順 | ローカル → クラウド → ローカルLLM の順 |

### 全体図

```
Phase 1  ローカルN-gram基盤
    ↓  CLIで変換が動く
Phase 2  OS統合 + 辞書・傾向DB
    ↓  IMEとして日常利用できるレベル
Phase 3  入力モード骨格（推論抽象レイヤー）
    ↓  モード切替の器が完成
Phase 4  クラウドLLM（パフォーマンスモード）
    ↓  主要差別化機能が動く
Phase 5  センシティブ検出 + ハイブリッドモード
    ↓  プライバシー保護が完成
Phase 5.5  再推論機能（右側文脈 + 選択再変換）
    ↓  変換精度の自己修正が動く
Phase 6  ローカルLLM（プロモード）
    ↓  v1 全機能が揃う ← v1 リリース
Phase 7  LoRA個人適応
         v2以降
```

## 13. リスクと対策

| リスク | 影響度 | 対策 |
|--------|--------|------|
| クラウドレイテンシが300msを超える | 高 | S-002で早期実測。NG時はローカルN-gram結果で代替 |
| センシティブフィールドの検出漏れ | 高 | 判定不能時は必ずローカル処理にフォールバック（S-003で網羅性確認） |
| プロンプト設計の品質が変換精度に直結する | 高 | Phase 4でS-001を最優先で実施 |
| OS統合（TSF/IMKitchen）の工数超過 | 高 | Phase 2でスタブ実装から開始し早期に結合を確認する |
| 右側文脈再推論の誤更新がUXを損なう | 高 | S-005で誤爆率を実測。閾値Dを保守的に設定。サイレント更新で目立たなくする |
| 再推論のクラウドAPIコストが想定超過 | 中 | バッチ化・低確信度トークン限定で抑制。ローカルLLMモード推奨を案内 |
| Workers AIの提供モデルが日本語に不十分 | 中 | ローカルLLMフォールバックで対応 |
| ローカルLLMとLM Studioのメモリ競合 | 中 | Ollama API方式を選択肢として提供し二重ロードを回避 |
| 文字数が変わる再推論でカーソル位置がズレる | 中 | v1では文字数が変わらない場合のみ更新。文字数変動対応はv2以降 |
| 日本語N-gramモデルの学習データ品質 | 中 | Wikipedia + CC-100の組み合わせで品質を担保（S-004で検証） |
| LoRAアダプタ更新のバックグラウンド処理がUXに影響 | 低 | 夜間バッチに限定。手動トリガーも提供 |
